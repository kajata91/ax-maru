# AX-Maru 2단계(자동화) 전환 설계

> 이 문서는 1단계 MVP(Claude Desktop Cowork 수동/예약 실행)에서 2단계(무인 상시 자동화)로 넘어가기 위한 **설계 문서**입니다.
> 코드 구현 전, 기획자/PM이 읽고 아키텍처·도구·비용·로드맵을 의사결정하기 위한 것입니다.
> 관련: [`CLAUDE.md`](CLAUDE.md) §2·§3·§6·§8·§9·§11 / [`AX-Maru_MVP_실행가이드.md`](AX-Maru_MVP_실행가이드.md) / [`자동화_설치가이드.md`](자동화_설치가이드.md)

> ⚠️ **채택 결정(2026-06-15): 무료 구성.** 이 문서가 권고한 유료 클라우드 큐레이션(방안 C/D + 3.1-1/3.1-2)은
> **채택하지 않았습니다.** 사용자 선택은 "수집은 Cowork 수동(무료), 자동화는 검증·공개·이메일(무료)"입니다.
> 실제 구현은 `.github/workflows/publish.yml` + `scripts/verify.mjs`이며, 설치는 `자동화_설치가이드.md` 참고.
> **비용 발생 옵션(매일 무인 자동 수집 등)은 실행 전 반드시 사용자 승인.** 아래 내용은 향후 유료 전환 시 참고용.

## 1. 목표와 전환 신호

### 1.1 2단계 목표 (한 줄 정의)

> **사람이 손대지 않아도, 매일 밤 자동 수집 → 아침 8시에 대시보드가 갱신되고 이메일 다이제스트가 도착한다.**

구체 목표(측정 가능한 형태):

| # | 목표 | 성공 기준(SLA) |
|---|------|----------------|
| G1 | 무인 상시 실행 | 사람 개입 0회로 7일 연속 산출 성공 |
| G2 | 야간 자동 수집 | 매일 **23:30~06:00 KST** 사이 수집·가공 완료 |
| G3 | 아침 대시보드 갱신 | 매일 **08:00 KST**까지 `site/`에 당일 데이터 반영 |
| G4 | 이메일 다이제스트 | 08:00 KST 전후 **kajata91@gmail.com**으로 발송 |
| G5 | 원문/번역 토글 | 대시보드 카드에서 영어 원문 ↔ 한국어 번역 전환 |
| G6 | 품질 유지 | 1단계 규칙(하루 20건 이내·1,000자 이내·출처 필수·날조 금지) 그대로 준수 |

> 품질 원칙은 1단계와 동일하다(**품질 > 속도**). 자동화는 "더 많이"가 아니라 "사람 없이도 같은 품질"을 위한 것이다.

### 1.2 전환 신호 (언제 넘어가나)

[`CLAUDE.md`](CLAUDE.md) §2 / MVP 가이드 §D의 신호를 그대로 따른다. **아래 중 하나라도 강해지면 전환:**

- [ ] 컴퓨터를 끈 밤사이에도 자동 수집이 필요하다 (수동 실행을 자꾸 까먹거나, 야간 발행 소식을 놓친다)
- [ ] 08시 정각 대시보드 + 이메일 자동 발송이 업무 루틴에 꼭 필요해졌다
- [ ] 원문/번역 토글·웹 UI 요구가 생겼다
- [ ] **운영 부담 신호:** 1단계를 1~2주 이상 매일 돌려봐서 지시문·품질이 안정됐고, 이제 "손이 가는 것" 자체가 병목이다

> 권고: **1단계 다이제스트 품질이 안정될 때까지는 2단계로 가지 않는다.** 자동화는 불안정한 프로세스를 더 빠르게 망가뜨린다. 품질 검증(지시문 튜닝)이 끝난 뒤 자동화한다.

## 2. 전체 아키텍처와 데이터 흐름

### 2.1 파이프라인 개요

```
[스케줄러]                  매일 23:30 KST 트리거
   │
   ▼
[1. 수집 에이전트] ── 웹 검색(Tier-1 소스) → 최근 24~72h 후보 풀 수집
   │                  산출: data/raw/YYYY-MM-DD.candidates.json (원시 후보)
   ▼
[2. 선별·가공]   ── 클러스터링·중요도·20건 엄선 → 한국어 요약+시사점(1,000자)
   │                  원문(en) + 번역(ko) 동시 보존 (G5용)
   ▼
[3. 산출]        ── (a) Daily/AX-Maru_Daily_YYYY-MM-DD.md  (Obsidian 카드)
   │                  (b) site/data/YYYY-MM-DD.json         (프론트 데이터)
   │                  (c) site/data/index.json 갱신          (날짜 목록)
   ▼
[4. 대시보드 갱신] ── site/ 정적 배포 (커밋&푸시 또는 호스팅 재배포)
   │                  목표: 08:00 KST 이전 반영
   ▼
[5. 이메일 발송]  ── 당일 다이제스트 요약 HTML → kajata91@gmail.com (08:00 KST)
   │
   ▼
[로그/알림]       ── 성공/실패 로그, 실패 시 폴백 + 알림
```

### 2.2 데이터 흐름 핵심

- **단일 소스 오브 트루스(SSOT)는 JSON.** `site/data/YYYY-MM-DD.json`을 먼저 만들고, Markdown 카드(`Daily/*.md`)와 이메일 본문은 그 JSON에서 렌더링한다. → 세 산출물이 절대 어긋나지 않는다.
- 각 항목 JSON 스키마(권고, 원문/번역 토글 G5 지원):
  ```json
  {
    "id": "2026-06-15-001",
    "importance": "상",            // 상/중/하
    "field": "AI/AX Hot News",     // §4 새 명칭 6분야
    "title_ko": "한국어 제목",
    "title_en": "Original title",
    "summary_ko": "요약+💡시사점 (1,000자 이내)",
    "summary_en": "원문 발췌/원문 요약 (영어)",
    "sources": [{"name":"Anthropic News","url":"...","published":"2026-06-14"}],
    "cluster": ["url2","url3"]     // 같은 사건 복수 출처
  }
  ```
- 메타: 문서 상단 (a)날짜 (b)총 건수 (c)이번 주 키워드 Top5 는 JSON `meta` 블록에 둔다.

### 2.3 스케줄러: 세 가지 방안 비교

핵심 질문: **"밤 23:30, 컴퓨터가 꺼져 있거나 잠들어 있어도 도는가?"** 이게 2단계의 본질(G2)이다.

| 방안 | 동작 위치 | 컴퓨터 꺼져도 도나 | 장점 | 단점 | 추천도 |
|------|-----------|--------------------|------|------|--------|
| **A. macOS launchd** | 내 맥 로컬 | ✕ (맥이 깨어 있어야 함) | 설정 단순, 추가 비용 0, 로컬 파일 직접 접근 | 맥이 꺼지면 실패 = G2 본질 미충족. `pmset`로 깨우기 가능하나 불안정 | △ |
| **B. cron (로컬)** | 내 맥 로컬 | ✕ | 가장 익숙 | macOS에선 launchd가 정석. A와 동일 한계 | ✕ |
| **C. Claude Code 스케줄(cron/scheduled task) — 클라우드 실행** | 원격(클라우드) | ○ | **맥 상태와 무관하게 실행**, 에이전트가 직접 웹검색·파일생성·git push·이메일까지 한 번에 | 원격 환경이 내 Obsidian Vault 로컬 경로에 직접 못 씀 → git 경유 필요. 도구 권한·비용 확인 필요 | ◎ |
| **D. GitHub Actions(cron)** | GitHub 클라우드 | ○ | 무료 한도 넉넉, 로그·재시도·시크릿 관리 성숙 | 수집 "지능"(LLM 큐레이션)은 별도 API 호출 필요(워크플로 안에서 Anthropic API 호출) | ○ (보조/대안) |

> **결론(권고): C(Claude Code 스케줄)를 1차, D(GitHub Actions)를 폴백/대안으로.**
> - 로컬 launchd(A/B)는 "밤사이 자동 수집"(G2)이라는 핵심 목표를 구조적으로 만족하지 못한다(맥이 깨어 있어야 함). 1단계의 한계를 그대로 물려받는다.
> - C는 에이전트가 클라우드에서 깨어나 **수집~이메일까지 엔드투엔드**로 수행할 수 있어 가장 적합. 단, 원격 실행은 로컬 Vault 경로에 직접 못 쓰므로 **결과를 git push → 로컬은 Obsidian Git 플러그인 등으로 pull** 하는 동기화 설계가 필요(§3.4).
> - D는 LLM 큐레이션 로직만 스크립트(`curate.mjs`)로 떼어내면 안정적인 무료 인프라가 된다. C가 비용/권한 이슈가 있으면 D로 대체.

#### KST/타임존 주의
- 스케줄은 보통 UTC 기준이다. **23:30 KST = 14:30 UTC**, **08:00 KST = 23:00 UTC(전날)**. 크론 표현식에 타임존을 명시하거나 UTC로 환산해 등록한다. 서머타임 없는 KST라 환산은 고정.

## 3. 단계별 구현 옵션과 트레이드오프

### 3.1 수집 자동화 (웹 검색 → 후보 풀)

| 옵션 | 방식 | 비용 | 신뢰성 | 비고 |
|------|------|------|--------|------|
| **3.1-1 Claude Code 에이전트 + 내장 웹검색** | 스케줄 실행 시 에이전트가 직접 Tier-1 소스 검색 | Claude 사용량 | 높음(에이전트가 판단·재시도) | **권고.** 1단계 Cowork 지시문을 거의 그대로 재사용 |
| 3.1-2 Anthropic API + 웹검색 도구(`curate.mjs`) | 스크립트가 API 호출, web search tool 사용 | API 토큰 비용 | 중~높음 | GitHub Actions(D)와 궁합. 이미 `scripts/package.json`에 `@anthropic-ai/sdk` 의존성 존재 |
| 3.1-3 RSS/사이트 피드 직접 파싱 | 소스별 RSS 수집 후 LLM은 선별만 | 낮음 | 소스 커버리지 한계 | 비용 최소지만 Tier-1 중 RSS 없는 곳 누락. 보조용 |

> 권고: **3.1-1(에이전트 직접 검색)**. 1단계에서 검증된 지시문 자산을 그대로 쓰는 게 가장 빠르고 품질 일관성이 높다. 비용 민감하면 3.1-3로 후보를 좁힌 뒤 LLM 선별만 시키는 하이브리드.

### 3.2 가공·선별

- 로직은 1단계 지시문([`AX-Maru_MVP_실행가이드.md`](AX-Maru_MVP_실행가이드.md) B) §1~5 그대로: 클러스터링 → 20건 엄선 → 1,000자 요약+시사점 → 중요도.
- **추가 산출(2단계 신규):** 원문/번역 토글(G5)을 위해 `summary_en`(영어)도 함께 생성·저장.
- **자동화 환경에서의 품질 가드레일:** 사람이 검수하지 않으므로 산출 후 자동 검증 스텝을 둔다(§5.2 검증 체크).

### 3.3 산출물 생성 (JSON/Markdown)

- 순서: **JSON 먼저 → 그로부터 MD·이메일 렌더**(§2.2 SSOT 원칙).
- `index.json`(날짜 목록) 갱신은 [`CLAUDE.md`](CLAUDE.md) §9-4 규칙 준수.

### 3.4 대시보드 배포

| 옵션 | 방식 | 비용 | 관리부담 | 비고 |
|------|------|------|----------|------|
| **3.4-1 GitHub Pages** | `site/`를 Pages로 서빙, 커밋 시 자동 배포 | 무료 | 낮음 | **권고.** 이미 GitHub(kajata91/ax-maru) 연동됨 |
| 3.4-2 Cloudflare Pages / Netlify / Vercel | 정적 호스팅 + git 연동 | 무료 한도 | 낮음 | 커스텀 도메인·빌드 유연. Pages로 부족할 때 |
| 3.4-3 로컬 Obsidian만 | 배포 없이 Vault 안에서만 열람 | 0 | 0 | 웹 UI·원문 토글(G5) 요구를 못 채움 |

- **로컬 Vault 동기화:** 원격(C/D)에서 생성한 `Daily/*.md`·`site/data/*`를 git push → 로컬 Obsidian은 **Obsidian Git 플러그인**으로 주기 pull. 이로써 "원격이 만든 결과를 내 Obsidian에서도 본다"가 성립.

### 3.5 이메일 발송

| 옵션 | 방식 | 비용 | 신뢰성 | 비고 |
|------|------|------|--------|------|
| **3.5-1 Gmail API (자기 계정 → 자기 계정)** | OAuth/앱 비밀번호로 kajata91 계정에서 자신에게 발송 | 무료 | 높음 | **권고.** 수신자=발신자=본인이라 스팸·인증 이슈 최소 |

## 5. 보안과 실패 대응

### 5.1 자격증명·보안

- **절대 커밋 금지:** API 키(Anthropic), Gmail 앱 비밀번호/OAuth 토큰. → `.env`에 두고 `.gitignore`에 추가.
  - 현재 `.gitignore`는 `.DS_Store`만 있음. **추가 필요:** `.env`, `scripts/.env`, `*.local`, `scripts/node_modules/`, `data/raw/`(원시 후보, 선택).
- **GitHub Actions(D) 사용 시:** 키는 코드가 아니라 **Repository Secrets**에 저장(`ANTHROPIC_API_KEY`, `GMAIL_APP_PASSWORD` 등).
- **Claude Code 스케줄(C) 사용 시:** 도구·커넥터 권한 범위를 최소화(필요한 것만), git push 자격은 기존 SSH 키(`~/.ssh/id_ed25519`) 재사용.
- **Gmail은 앱 비밀번호 권장**(계정 비밀번호 직접 사용 금지, 2단계 인증 활성 전제). 권한은 발송(send)만.
- 산출 JSON/MD에는 자격증명·내부 경로가 절대 들어가지 않게 렌더 단계에서 가드.

### 5.2 실패 대응(폴백·알림)

| 실패 지점 | 폴백 | 알림 |
|-----------|------|------|
| 수집 0건 / 웹검색 실패 | 재시도(지수 백오프 N회) → 그래도 실패면 **"오늘 자동 수집 실패" 마커 JSON** 생성(빈 다이제스트 아님) | 실패 메일/푸시 |
| 선별·가공 LLM 오류 | 부분 결과라도 저장 + 실패 항목 표시 | 경고 메일 |
| 검증(`verify.mjs`) 실패(20건 초과·1,000자 초과·링크 깨짐) | 자동 보정 가능하면 보정, 아니면 발행 보류·초안 보관 | 경고 메일 |
| 대시보드 배포 실패 | git push 재시도, 직전 버전 유지(롤백 안전) | 실패 메일 |
| 이메일 발송 실패 | 재시도 → 실패해도 대시보드는 갱신됨(부분 성공 허용) | 로그 + 다음날 합산 알림 |

- **공통 원칙:** 각 단계는 **멱등(idempotent)**·**부분 성공 허용**. 한 단계 실패가 전체를 막지 않게 단계 분리.
- **로그:** `scripts/logs/YYYY-MM-DD.log` 또는 stdout → 스케줄러 로그. 성공/실패 1줄 요약을 [`memory.md`](memory.md)에 누적([`CLAUDE.md`](CLAUDE.md) §9-2)할지 여부는 결정 필요(§7).
